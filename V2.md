## 2. Layer Architecture

VyroEngine follows a **strict layered architecture**. Higher layers may depend on lower layers. Lower layers must never depend on higher layers.

```
Layer 7: Game Code / Scripts
    ↓ depends on
Layer 6: Editor / Tools
    ↓ depends on
Layer 5: High-Level Systems (Scene, Gameplay)
    ↓ depends on
Layer 4: Mid-Level Systems (Rendering, Physics, Audio)
    ↓ depends on
Layer 3: Core Systems (ECS, Events, Input, Memory)
    ↓ depends on
Layer 2: Platform Abstraction Layer (PAL)
    ↓ depends on
Layer 1: OS / Hardware
```

### 2.1 Layer Dependency Rules

- `#include` of a higher layer from a lower layer is a **build error** (enforced via CMake targets)
- Circular dependencies between modules in the same layer are **forbidden**
- All cross-layer communication uses **interfaces** (abstract base classes or concepts)

## 3. Core Engine Architecture

### 3.1 Application Lifecycle

```
main()
  └── Application::Create()
        ├── Platform::Init()         // Window, file system, threading
        ├── Logger::Init()           // Logging subsystem
        ├── MemoryManager::Init()    // Allocator setup
        ├── EventBus::Init()         // Event system
        ├── InputSystem::Init()      // Input mapping
        ├── AssetManager::Init()     // Asset registry
        ├── Renderer::Init()         // GAL + render pipeline
        ├── PhysicsWorld::Init()     // Physics simulation
        ├── AudioEngine::Init()      // Audio device
        ├── AnimationSystem::Init()  // Animation runtime
        ├── ScriptEngine::Init()     // Lua VM
        └── Application::Run()
              ├── [Game Loop]
              │     ├── PollEvents()
              │     ├── Update(dt)
              │     │     ├── InputSystem::Update()
              │     │     ├── ScriptEngine::Update()
              │     │     ├── PhysicsWorld::Step()
              │     │     ├── AnimationSystem::Update()
              │     │     └── ECS::RunSystems()
              │     └── Render()
              │           ├── Renderer::BeginFrame()
              │           ├── Renderer::Submit(scene)
              │           └── Renderer::EndFrame()
              └── Application::Shutdown()
```

### 3.2 Game Loop

```cpp
// Fixed timestep with variable rendering
constexpr float FIXED_TIMESTEP = 1.0f / 60.0f;
float accumulator = 0.0f;

while (running) {
    float dt = timer.Delta();
    accumulator += dt;

    PollEvents();

    while (accumulator >= FIXED_TIMESTEP) {
        FixedUpdate(FIXED_TIMESTEP);   // Physics, deterministic logic
        accumulator -= FIXED_TIMESTEP;
    }

    float alpha = accumulator / FIXED_TIMESTEP;  // Interpolation factor
    VariableUpdate(dt);     // Input, scripts, animation
    Render(alpha);          // Interpolated rendering
}
```

### 3.3 Event System Architecture

```
EventBus (Central)
  ├── Publish<EventType>(data)     → immediate or queued dispatch
  ├── Subscribe<EventType>(fn)     → callback registration
  ├── Unsubscribe(handle)          → cleanup
  └── DispatchQueue()              → process queued events

Event Types:
  ├── WindowResizeEvent{w, h}
  ├── KeyPressEvent{keycode, mods}
  ├── KeyReleaseEvent{keycode}
  ├── MouseMoveEvent{x, y, dx, dy}
  ├── MouseButtonEvent{button, action}
  ├── ScrollEvent{dx, dy}
  ├── EntityCreatedEvent{id}
  ├── EntityDestroyedEvent{id}
  ├── ComponentAddedEvent{id, type}
  ├── AssetLoadedEvent{uuid}
  └── SceneChangedEvent{scene_id}
```

### 3.4 Logging System

```
Logger (spdlog-backed)
  ├── Sinks:
  │     ├── ConsoleSink (colored terminal)
  │     ├── FileSink (rotating log files)
  │     ├── EditorSink (ImGui console panel)
  │     └── DebugSink (OS debug output)
  ├── Levels: TRACE, DEBUG, INFO, WARN, ERROR, CRITICAL
  ├── Channels: ENGINE, RENDERER, PHYSICS, AUDIO, SCRIPT, GAME
  └── Format: [timestamp][channel][level] message {context}

Macros:
  VYRO_TRACE(...)    → channel-aware trace
  VYRO_INFO(...)     → info level
  VYRO_WARN(...)     → warning
  VYRO_ERROR(...)    → error
  VYRO_ASSERT(x)     → debug assert (fatal in debug, no-op in release)
  VYRO_VERIFY(x)     → release assert (always active)
```

## 4. ECS Architecture

### 4.1 Design Overview

VyroEngine uses an **Archetype-based ECS** — the same pattern as Unity DOTS, Bevy, and Flecs. Entities with the same component signature share the same archetype (contiguous memory block), enabling cache-optimal iteration.

```
Entity = uint64_t (32-bit index | 16-bit version | 16-bit flags)

Archetype Table:
  ┌─────────────────────────────────────────┐
  │ Archetype: [Transform, Mesh, Material]  │
  ├────────────┬──────────┬─────────────────┤
  │ Transform[] │ Mesh[]   │ Material[]      │
  │ [0]         │ [0]      │ [0]             │
  │ [1]         │ [1]      │ [1]             │
  │ [2]         │ [2]      │ [2]             │
  └────────────┴──────────┴─────────────────┘
  (all arrays are contiguous, same index = same entity)
```

### 4.2 Core ECS Types

```cpp
// Entity handle
using EntityID = uint64_t;
constexpr EntityID NULL_ENTITY = 0;

// Component concept
template<typename T>
concept Component = std::is_trivially_copyable_v<T>;

// System interface
struct ISystem {
    virtual void OnUpdate(float dt) = 0;
    virtual void OnFixedUpdate(float dt) {}
    virtual SystemPriority Priority() const { return 0; }
};

// Query — zero-overhead archetype iteration
auto query = world.Query<Transform, Velocity>()
                  .Exclude<StaticTag>()
                  .Build();

query.ForEach([](Transform& t, const Velocity& v) {
    t.position += v.linear * dt;
});
```

### 4.3 Scene Graph

```
Scene
  ├── EntityRegistry (ECS world)
  ├── SceneGraph (parent-child transforms)
  │     ├── Node {entity_id, local_transform, world_transform}
  │     └── DirtyFlags (lazy world transform recompute)
  ├── Systems (ordered execution)
  └── Resources (scene-local singletons)
```

## 5. Rendering Architecture

### 5.1 Render Graph

VyroEngine uses a **frame graph (render graph)** — a directed acyclic graph of render passes that automatically handles resource transitions, synchronization, and GPU barriers.

```
RenderGraph (per frame)
  ├── Pass: ShadowMap
  │     ├── Input: scene geometry
  │     └── Output: shadow_depth_texture
  ├── Pass: GBuffer
  │     ├── Input: scene geometry, materials
  │     └── Output: albedo_rt, normal_rt, metallic_roughness_rt, depth_rt
  ├── Pass: SSAO
  │     ├── Input: normal_rt, depth_rt
  │     └── Output: ssao_texture
  ├── Pass: Lighting
  │     ├── Input: GBuffer RTs, shadow_depth, ssao_texture, lights
  │     └── Output: hdr_color_rt
  ├── Pass: Skybox
  │     ├── Input: hdr_color_rt, cubemap
  │     └── Output: hdr_color_rt
  ├── Pass: PostProcess
  │     ├── Input: hdr_color_rt
  │     └── Output: ldr_color_rt (tonemapped)
  └── Pass: UI
        ├── Input: ldr_color_rt, ui_draw_data
        └── Output: swapchain image
```

### 5.2 Graphics Abstraction Layer (GAL)

```cpp
// Backend-agnostic resource types
struct BufferHandle    { uint32_t id; };
struct TextureHandle   { uint32_t id; };
struct PipelineHandle  { uint32_t id; };
struct ShaderHandle    { uint32_t id; };
struct SamplerHandle   { uint32_t id; };
struct RenderPassHandle{ uint32_t id; };

// GAL interface (implemented by Vulkan/OpenGL/Metal backends)
class IRenderDevice {
    virtual BufferHandle  CreateBuffer(const BufferDesc&)  = 0;
    virtual TextureHandle CreateTexture(const TextureDesc&) = 0;
    virtual PipelineHandle CreatePipeline(const PipelineDesc&) = 0;
    virtual void SubmitCommandBuffer(CommandBuffer&) = 0;
    virtual void Present() = 0;
};
```

### 5.3 Material System

```
Material
  ├── Shader (vertex + fragment + optional: geometry, compute)
  ├── Parameters (UBO data: albedo, roughness, metallic, emissive)
  ├── Textures (bound texture slots)
  └── RenderState (blend mode, depth test, cull mode, fill mode)

Material Instances
  └── Share shader, override parameters per-instance (UBO offset)
```

### 5.4 2D Renderer

```
Renderer2D
  ├── SpriteBatch (dynamic VBO, up to 10,000 quads per batch)
  ├── TextureAtlas manager
  ├── Font renderer (SDF fonts via msdf-atlas-gen)
  ├── Primitive renderer (lines, circles, rects)
  └── Sort layers (z-order, material, texture)
```

### 5.5 Vulkan Architecture (Phase 10)

```
VulkanDevice
  ├── Instance + Debug Messenger
  ├── Physical Device (GPU selection with scoring)
  ├── Logical Device (queues: graphics, compute, transfer, present)
  ├── Swapchain (triple-buffered)
  ├── Command Pools (per-thread)
  ├── Descriptor Pool + Cache
  ├── Pipeline Cache (disk-persisted)
  └── VMA (memory allocator)

Frame-in-flight: 3 frames
  └── Per-frame: Command Buffer, Semaphores, Fences, Uniform Buffers
```

## 6. Physics Architecture

### 6.1 Overview

```
PhysicsWorld
  ├── BroadPhase (spatial partitioning: BVH / dynamic AABB tree)
  ├── NarrowPhase (GJK + EPA for convex shapes, SAT for AABB)
  ├── CollisionManifold (contact points, normals, depth)
  ├── ConstraintSolver (sequential impulse, Baumgarte stabilization)
  ├── RigidBodyManager (integration, forces, torques)
  └── PhysicsDebugRenderer (debug overlay)
```

### 6.2 Simulation Loop

```
PhysicsWorld::Step(dt = 1/60s)
  ├── ApplyForces()              // gravity, user forces
  ├── IntegrateVelocities()      // semi-implicit Euler
  ├── BroadPhase::Update()       // AABB overlaps
  ├── NarrowPhase::Detect()      // exact contacts
  ├── BuildConstraints()         // contact + user constraints
  ├── SolveConstraints(iter=10)  // sequential impulse
  └── IntegratePositions()       // update transforms
```

## 10. Asset Pipeline Architecture

```
Asset UUID → AssetRegistry → AssetLoader → Runtime Asset

AssetRegistry
  ├── UUID → file path mapping (stored in .vyro_project)
  ├── Metadata cache (type, dependencies, last modified)
  └── Dependency tracking (asset A uses asset B)

AssetLoaders (registered by type):
  ├── TextureLoader   → stb_image → Texture2D
  ├── MeshLoader      → Assimp → Mesh + Materials
  ├── ShaderLoader    → glslc → SPIRV binary
  ├── AudioLoader     → miniaudio → AudioBuffer
  ├── FontLoader      → msdf-atlas-gen → FontAtlas
  ├── SceneLoader     → JSON → Scene
  └── ScriptLoader    → Lua source → ByteCode (LuaJIT)

Asset Hot Reload:
  └── FileWatcher → detect change → reload → broadcast AssetLoadedEvent
```

## 11. Memory Architecture

### 11.1 Allocator Hierarchy

```
Global Heap
  ├── PoolAllocator    — fixed-size objects (components, entities)
  │     └── 64B, 128B, 256B, 512B, 1KB pools
  ├── StackAllocator   — per-frame temporary data (render commands)
  │     └── Reset at frame end
  ├── ArenaAllocator   — per-system bulk allocation
  │     └── Reset at system boundary
  └── FreeListAllocator — general-purpose with coalescing

Component Storage:
  └── Archetype tables use aligned, contiguous arrays (64-byte aligned)
      for cache-line-friendly iteration

Debug Tracking:
  └── Debug builds wrap all allocators with:
      ├── Allocation size tracking
      ├── Use-after-free detection (guard pages)
      └── Leak reporting on shutdown
```

### 11.2 GPU Memory (Vulkan)

```
VMA (Vulkan Memory Allocator)
  ├── VK_MEMORY_PROPERTY_DEVICE_LOCAL_BIT  → GPU-only (meshes, textures)
  ├── VK_MEMORY_PROPERTY_HOST_VISIBLE_BIT  → CPU-GPU shared (uniforms)
  └── Staging buffers for transfers

Budget tracking:
  ├── Per-heap budget monitoring
  └── Eviction strategy for texture streaming
```

## 12. Networking Architecture

```
NetworkManager
  ├── Transport Layer
  │     ├── Reliable UDP (custom or ENet)
  │     └── TCP (optional, for lobby/auth)
  ├── Server
  │     ├── EntityReplicator (which entities sync to which clients)
  │     ├── SnapshotManager (game state snapshots for rollback)
  │     └── AuthorityController (server-authoritative or P2P)
  ├── Client
  │     ├── InputPrediction (local prediction, server correction)
  │     ├── LagCompensation (rewind-based hit detection)
  │     └── StateInterpolation (smooth entity movement)
  └── Serialization
        ├── BitStream (compact binary protocol)
        └── DeltaCompression (only changed fields)
```

## 13. Threading Model

```
Main Thread:
  ├── Game logic (ECS update)
  ├── Editor UI (ImGui)
  └── Render command building

Render Thread:
  ├── GPU command submission
  ├── Swap chain presentation
  └── Resource uploads

Physics Thread:
  └── Fixed-timestep physics simulation

Audio Thread:
  └── Audio mixing + DSP (runs at audio callback rate)

Worker Threads (job system):
  ├── Asset loading/decompression
  ├── Particle system updates
  ├── Animation bone transforms
  └── Occlusion culling

Job System:
  ├── Lock-free task queue (MPMC ring buffer)
  ├── Work stealing between workers
  └── Fiber-based continuation (future)
```
