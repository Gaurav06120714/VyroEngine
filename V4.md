## Phase 7: Scripting System

### Goal
A full Lua scripting integration with a comprehensive engine API, hot reload, and async coroutine support.

### Why It Is Needed
Hard-coding game logic in C++ means every gameplay change requires recompile. A scripting system enables rapid iteration, moddability, and separates engine code from game code. Lua is the industry standard for embedded game scripting.

**Industry Example:** Roblox is 100% Lua. World of Warcraft uses Lua for addons. Corona SDK (now Solar2D) uses Lua. Godot uses GDScript (Lua-inspired).

### Estimated Duration
**5–6 weeks**

### Prerequisites
- Phase 1 (asset manager — `ScriptAsset`)
- Phase 2 (ECS — `ScriptComponent`)
- Phase 3 (renderer — expose rendering debug API)
- Phase 4 (physics — expose raycast API)

---

### Sub-Phase 7.1: Lua Integration

#### Goal
Embed LuaJIT, bind the engine API, and execute scripts from disk.

#### Duration: 7 days

#### Deliverables
- LuaJIT 2.1 embedded via CMake
- sol3 3.3+ binding layer
- `ScriptEngine` — manages LuaJIT state(s)
- Per-entity sandbox (each script gets isolated environment)
- Safe error handling (Lua errors don't crash engine)
- Script asset type (`*.lua`) integrated with AssetManager
- Script lifecycle: `OnCreate()`, `OnUpdate(dt)`, `OnDestroy()`

#### Technical Challenges
- Sandboxing (preventing scripts from accessing system I/O)
- Error propagation (Lua errors → engine error log)
- sol3 binding performance overhead

#### Success Criteria
- Lua script moves an entity's transform by component access
- Lua runtime error prints stack trace to console without crashing

---

### Sub-Phase 7.2: Script API

#### Goal
Expose all major engine systems to Lua scripts.

#### Duration: 12 days

#### Deliverables

```lua
-- API targets
-- Entity access
local entity = Entity.Create()
local transform = entity:GetComponent(Transform)
transform.position = Vec3(1, 2, 3)
entity:AddComponent(RigidBody, { mass = 1.0 })
entity:Destroy()

-- Input
if Input.IsKeyDown(Key.Space) then ... end
local h = Input.GetAxis("Horizontal")

-- Audio
Audio.Play("explosion.ogg", entity.position, { volume = 0.8 })

-- Physics
local hit = Physics.Raycast(origin, direction, maxDist)
if hit then print(hit.entity, hit.normal) end

-- Scene
local other = Scene.FindByName("Player")
Scene.LoadAdditive("level2.vscene")

-- Time
local dt = Time.DeltaTime
local t  = Time.Elapsed
```

Full binding list:
- `Vec2`, `Vec3`, `Vec4`, `Quat`, `Mat4` math types
- `Entity`, `World` — ECS access
- `Transform`, `RigidBody`, `Collider`, `AudioSource` components
- `Input` — keyboard, mouse, gamepad
- `Physics` — raycast, overlap queries
- `Audio` — play, stop, adjust
- `Scene` — find, load, unload
- `Time` — delta, elapsed, scale
- `EventBus` — subscribe/publish from Lua
- `Debug` — draw lines, spheres, text

#### Success Criteria
- A complete Pong game implementable in pure Lua
- All API calls have error messages for wrong argument types

---

### Sub-Phase 7.3: Hot Reloading

#### Goal
Reload changed Lua scripts at runtime without restarting the engine or losing game state.

#### Duration: 5 days

#### Deliverables
- File watcher integration for `*.lua` files
- Script reload strategy: re-execute module, re-call `OnCreate()` if needed
- State preservation hints (via `OnSave()` / `OnRestore()` callbacks)
- Reload notification in editor console
- Coroutine restart on reload

#### Success Criteria
- Changing a script file reloads it within 300ms
- In-flight coroutines gracefully handle reload

---

### Phase 7 Skills Required
- Lua 5.4 / LuaJIT internals
- sol3 C++ binding
- Sandboxing and security

### Phase 7 Success Criteria
- Complete Pong or Breakout implemented in Lua
- Hot reload works without engine restart

## Phase 8: Editor Development

### Goal
A fully functional level editor (VyroEditor) built on Dear ImGui, running the engine as a library.

### Why It Is Needed
Without an editor, every game scene must be hardcoded. The editor multiplies developer productivity by 10x. It is also the primary showcase of the engine's capabilities.

**Industry Example:** Unity Editor, Unreal Editor, Godot Editor. All major engines are editor-first.

### Estimated Duration
**10–14 weeks**

### Prerequisites
- All Phases 1–7 complete
- Dear ImGui 1.90+ integrated
- ImGuizmo integrated

---

### Sub-Phase 8.1: Editor Framework

#### Goal
The editor shell: main window, dockable panels, menu bar, toolbar.

#### Duration: 10 days

#### Deliverables
- `EditorApplication` — extends engine Application
- ImGui docking layout (panels dock anywhere)
- Main menu bar (File, Edit, View, Help)
- Toolbar (New, Save, Open, Play, Pause, Stop)
- Theme system (dark/light/custom)
- Editor preferences persistence (JSON)
- Editor layer vs game layer separation

#### Success Criteria
- Editor opens, panels dock/undock correctly
- Theme applies globally

---

### Sub-Phase 8.2: Scene Hierarchy

#### Goal
A tree-view panel showing all entities in the scene.

#### Duration: 7 days

#### Deliverables
- Tree view of entity hierarchy (parent/child)
- Right-click context menu: Create Entity, Duplicate, Delete
- Entity renaming (double-click)
- Drag-and-drop reparenting
- Multi-select (Ctrl+Click, Shift+Click)
- Entity visibility toggle (hide in editor)
- Entity lock toggle (prevent selection)
- Search/filter entities by name

#### Success Criteria
- 1000 entities displayed in hierarchy without lag
- Drag-and-drop reparenting reflects in scene graph

---

### Sub-Phase 8.3: Inspector

#### Goal
A properties panel that shows and edits components of the selected entity.

#### Duration: 10 days

#### Deliverables
- Component display for all built-in components
- Property editors: float, vec2, vec3, color, bool, string, enum
- Asset picker (drag-and-drop or file browser)
- Add Component button + searchable list
- Remove Component button (right-click)
- Component copy/paste
- Undo/redo for property changes
- Component order (drag to reorder)

#### Technical Challenges
- Reflection system for auto-generating inspector UI from struct fields
- Custom property drawers for complex types

#### Success Criteria
- Every built-in component editable in inspector
- Undo/redo works for all property changes

---

### Sub-Phase 8.4: Asset Browser

#### Goal
A file browser panel for project assets with preview and import support.

#### Duration: 8 days

#### Deliverables
- File system tree view (directories)
- Asset grid with icons and thumbnails
- Texture preview (thumbnail on hover)
- Right-click context: Import, Delete, Rename, Show in OS
- Asset drag-to-inspector (assign texture to material slot)
- Import dialog with per-asset settings
- Asset search

#### Success Criteria
- Drag a texture from asset browser onto material → assigns correctly
- Thumbnail generated for all image assets

---

### Sub-Phase 8.5: Gizmos

#### Goal
3D transform manipulation gizmos in the viewport.

#### Duration: 7 days

#### Deliverables
- ImGuizmo integration
- Translate gizmo (per-axis and plane handles)
- Rotate gizmo (per-axis and free rotate)
- Scale gizmo (uniform and per-axis)
- World/local space toggle
- Snap-to-grid (hold Ctrl)

- Selected entity highlighted (outline)

#### Success Criteria
- All gizmo operations feel responsive (< 1 frame lag)
- Snap grid configurable (0.1, 0.25, 0.5, 1.0 units)

### Phase 8 Skills Required
- Dear ImGui advanced usage (docking, tables, drag-drop)

## Phase 9: Networking

### Goal
Client-server multiplayer networking with entity replication and basic lag compensation.

### Why It Is Needed
Multiplayer is a core requirement for many modern games. Adding networking later is architectural nightmare — it must be designed early even if implemented late.

**Industry Example:** Unreal's NetDriver + replication system. Unity's NGO (Netcode for GameObjects). Godot's MultiplayerAPI.

### Estimated Duration
**8–10 weeks**

### Prerequisites
- Phase 2 (ECS — networked components need replication flags)
- Phase 1 (event system — network events)

---

### Sub-Phase 9.1: Client-Server Architecture

#### Duration: 10 days

#### Deliverables
- ENet reliable UDP transport layer
- `NetworkManager` — host/join/disconnect
- Server loop (dedicated or listen server)
- Client connection management
- Connection handshake + authentication placeholder
- `NetworkRole` enum: Server, Client, Host, None
- Network configuration (port, max clients, tickrate)

#### Success Criteria
- Two instances connect on local network
- Server sends heartbeat, client receives it

---

### Sub-Phase 9.2: Replication

#### Duration: 14 days

#### Deliverables
- `[Replicated]` component annotation
- Server-to-client entity snapshot
- `NetworkID` component (identifies entities across network)
- Spawn / destroy replication (server spawns, client receives)
- Component property replication (delta only)
- Relevancy system (only replicate entities near each client)
- Bandwidth budget enforcement

#### Technical Challenges
- Delta compression for component state
- Handling entity spawn ordering (dependencies)
- Relevancy computation at scale

#### Success Criteria
- Moving a box on server appears moving on client

---

### Sub-Phase 9.3: Synchronization

#### Duration: 10 days

#### Deliverables
- Client-side prediction (input applied immediately, corrected by server)
- State interpolation (smooth movement between server snapshots)
- Rollback-ready architecture (state snapshots stored for N ticks)
- Clock synchronization (NTP-style)
- Lag compensation for projectile hits
- Network statistics HUD (ping, packet loss, bandwidth)

#### Success Criteria
- Character movement feels responsive at 100ms ping
- No visible rubber-banding at typical latency

---

### Phase 9 Skills Required
- Network programming (UDP, sockets)
- Game network theory (client prediction, interpolation, rollback)
- ENet API

### Phase 9 Success Criteria
- 2-4 player networked demo running on LAN
- Ping < 100ms shows no visible desync

## Phase 10: Advanced Graphics

### Goal
Vulkan-backed rendering with PBR, cascaded shadows, SSAO, post-processing, and a foundation for global illumination.

### Why It Is Needed
Modern games require Vulkan for performance, PBR for visual quality, and post-processing for cinematic feel. This phase elevates VyroEngine from "tech demo" to "visually competitive."

**Industry Example:** Unreal's Lumen + Nanite. Unity's HDRP. Godot 4's Vulkan renderer.

### Estimated Duration
**14–18 weeks**

### Prerequisites
- Phase 3 (OpenGL renderer — GAL interface)
- Phase 2 (ECS)
- Vulkan SDK installed, Vulkan knowledge required

---

### Sub-Phase 10.1: Vulkan Renderer

#### Duration: 20 days

#### Deliverables
- `VulkanRenderDevice` implementing `IRenderDevice`
- Vulkan instance + validation layers
- Physical/logical device with queue families
- Triple-buffered swapchain
- Command buffer recording per frame
- VMA memory allocation
- Descriptor set management (bindless future-ready)
- Pipeline cache (disk-persisted)
- Dynamic rendering (VK_KHR_dynamic_rendering)
- SPIRV shader loading
- Multi-threaded command recording (one command pool per thread)

#### Technical Challenges
- Synchronization: semaphores, fences, pipeline barriers
- Resource transition tracking in render graph
- Descriptor set layout management

#### Success Criteria
- Full engine scene renders via Vulkan backend
- Zero Vulkan validation errors in debug mode

---

### Sub-Phase 10.2: PBR (Physically Based Rendering)

#### Duration: 14 days

#### Deliverables
- Cook-Torrance BRDF (GGX NDF, Smith-Schlick geometry, Fresnel-Schlick)
- Metallic-roughness workflow (glTF standard)
- IBL (Image-Based Lighting): diffuse irradiance + specular prefilter
- BRDF LUT precomputation
- Normal mapping
- Emissive materials
- HDR render target + tone mapping (ACES, Reinhard, Uncharted2)

#### Success Criteria
- Materials match reference renders from glTF Sample Viewer
- Metal sphere with roughness gradient looks physically correct

---

### Sub-Phase 10.3: Shadows

#### Duration: 10 days

#### Deliverables
- Cascaded Shadow Maps (CSM) — directional light, 4 cascades
- PCF soft shadows
- Point light shadow maps (cubemap shadows)
- Spot light shadow maps
- Shadow bias auto-adjustment
- Shadow distance fade

#### Success Criteria
- No visible shadow acne or peter-panning
- CSM transitions invisible at default settings

---

### Sub-Phase 10.4: Post Processing

#### Duration: 10 days

#### Deliverables
- Post-process stack (ordered effect chain)
- Bloom (threshold + blur, physically-based)
- SSAO (screen-space ambient occlusion)
- Color grading (LUT-based)
- Vignette
- Chromatic aberration
- Motion blur (per-object, velocity buffer)
- TAA (Temporal Anti-Aliasing)
- FXAA (fallback)
- Depth of Field (Bokeh)

#### Success Criteria
- Bloom, SSAO, TAA toggleable in editor without restart
- Post-process stack configurable per-camera

---

### Sub-Phase 10.5: Global Illumination

#### Duration: 14 days

#### Deliverables
- Irradiance Volumes (light probes, Lumen-lite approach)
- Reflection Captures (static cubemap captures)
- Screen-Space Reflections (SSR) — dynamic, limited range
- Voxel GI prototype (64³ voxel grid, low-res indirect lighting)

#### Technical Challenges
- Probe update frequency vs performance
- SSR artifacts at grazing angles
- Voxelization at 60fps

#### Success Criteria
- Indirect lighting visually correct on static scenes
- SSR reflects dynamic objects convincingly

---

### Phase 10 Skills Required
- Vulkan API (advanced)
- PBR theory and implementation
- GLSL compute shaders
- Shadow mapping algorithms (CSM, PCF)

### Phase 10 Success Criteria
- Demo scene: Sponza with full PBR, CSM, SSAO, bloom, TAA at 60fps

## Phase 11: Optimization

### Goal
Profile-driven optimization across all engine systems to hit performance targets on minimum hardware.

### Why It Is Needed
A feature-complete engine that runs at 20fps is unusable. Optimization must be a dedicated phase — premature optimization wastes time, but zero optimization ships an unusable engine.

**Industry Example:** Every AAA engine has dedicated optimization sprints. Unreal Insights, Unity Profiler, Godot's built-in profiler.

### Estimated Duration
**6–8 weeks**

### Prerequisites
- All Phases 1–10 complete
- Tracy profiler integrated

---

### Sub-Phase 11.1: Profiling

#### Duration: 7 days

#### Deliverables
- Tracy 0.10 integration (CPU + GPU profiling)
- `VYRO_PROFILE_SCOPE(name)` macro
- GPU timestamp queries
- Memory profiling (per-allocator tracking)
- Frame time graphs in editor
- Asset loading profiling
- Network bandwidth profiling

#### Success Criteria
- Every significant function has a Tracy zone
- Frame breakdown visible at per-system granularity

---

### Sub-Phase 11.2: Memory Optimization

#### Duration: 10 days

#### Deliverables
- Identify and eliminate allocation hotspots (Tracy allocation tracking)
- Component layout optimization (pack related data, eliminate padding)
- String interning system (eliminate duplicate strings)
- Asset streaming LRU cache (evict unused assets)
- GPU memory defragmentation (VMA defrag API)
- Memory budget reports in profiler

#### Success Criteria
- Base runtime under 100MB memory
- Zero per-frame heap allocations in hot path

---

### Sub-Phase 11.3: Rendering Optimization

#### Duration: 14 days

#### Deliverables
- GPU instancing (1 draw call per mesh type)
- Indirect draw (GPU-driven rendering preparation)
- Occlusion culling (GPU occlusion queries or HZB)
- LOD system (mesh LODs, shader LODs)
- Texture streaming (stream in/out based on distance)
- Multi-threaded command buffer recording
- Batch sorting by material to minimize state changes

#### Success Criteria
- 100,000 instanced objects at 60fps
- Draw call count < 3,000 on complex scene

---

### Phase 11 Skills Required
- Profiling methodology
- GPU optimization (GPU-driven rendering)
- Cache-friendly data layout

### Phase 11 Success Criteria
- Full demo scene (Phase 10 Sponza) at 60fps on minimum hardware (GTX 1060)
- Memory under budget
- Zero per-frame heap allocations

- Test suite green
- No P0 bugs open
