## Phase 0: Research & Planning

### Goal
Establish complete technical knowledge and design decisions before writing a single line of production code. Prevent architectural mistakes that would require full rewrites later.

### Why It Is Needed
Game engines built without planning become "big ball of mud" codebases. Phase 0 ensures every major decision (ECS type, rendering API, build system, scripting language) has been evaluated against alternatives, has documented trade-offs, and has a committed direction.

**Industry Example:** Unreal Engine 5 spent years on research before shipping Nanite and Lumen. Unity spent 3+ years designing DOTS before public release.

### Estimated Duration
**3–4 weeks** (solo developer)

### Prerequisites
- None (this is the first phase)

### Objectives
1. Study and document: Hazel Engine, Godot 4, Flecs, EnTT, Bevy ECS architectures
2. Evaluate rendering APIs: Vulkan vs DirectX 12 vs Metal vs WebGPU
3. Evaluate scripting: Lua vs Python vs JavaScript vs AngelScript
4. Evaluate physics: Bullet3 vs PhysX vs Jolt vs custom
5. Evaluate audio: OpenAL vs miniaudio vs FMOD vs Wwise
6. Design memory layout strategy
7. Define coding standards and style guide
8. Set up GitHub repository and issue tracking
9. Define milestone schedule

### Deliverables
- [ ] Architecture decision records (ADRs) for all major choices
- [ ] Technology evaluation matrix
- [ ] Coding standards document (→ rulz/)
- [ ] GitHub repository initialized
- [ ] CMake project template created
- [ ] Initial README.md
- [ ] Phase 1–12 task breakdown in GitHub Issues
- [ ] Development environment setup guide

### Risks
| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| Choosing wrong ECS architecture | Medium | High | Prototype both archetype and sparse-set; benchmark |
| Rendering API over-complexity | High | High | Start with OpenGL, add Vulkan in Phase 10 |
| Scope creep from day 1 | High | Medium | Strict phase gates, MVP-first mentality |
| Build system complexity | Low | Medium | Use CMake with presets from the start |

### Dependencies
- None

### Milestones
| Milestone | Target |
|-----------|--------|
| Technology stack finalized | Week 1 |
| Architecture documents complete | Week 2 |
| Coding standards published | Week 3 |
| Repository + CI skeleton ready | Week 4 |

### Success Criteria
- All ADRs written and reviewed
- Technology stack locked (no "maybe" items)
- CI pipeline running on empty project
- Coding standards enforced by Clang-Format config

## Phase 1: Core Engine Foundation

### Goal
Build the non-negotiable foundation that every other system depends on. A rock-solid core means every subsequent phase builds on stable ground.

### Why It Is Needed
Without logging, you cannot debug. Without memory management, you have allocator fragmentation. Without events, systems cannot communicate. Without input, you cannot test interactively. The core is the immune system of the engine.

**Industry Example:** Unreal Engine's `CoreUObject`, Godot's `core/` module, id Tech's `common.cpp` — all engines have a small but critically important core layer.

### Estimated Duration
**8–10 weeks** (solo developer)

### Prerequisites
- Phase 0 complete
- Development environment configured
- CMake + Clang + vcpkg/conan ready

---

### Sub-Phase 1.4: Event System

#### Goal
A type-safe, zero-overhead event bus supporting both synchronous and deferred event dispatch.

#### Duration: 5 days

#### Deliverables
- `EventBus` singleton with templated publish/subscribe
- `EventHandle` for subscription management
- Synchronous dispatch (immediate, inline)
- Deferred dispatch (queued, dispatched at frame boundary)
- Built-in engine events (window, keyboard, mouse)
- Performance: 1M events/sec minimum

#### Technical Challenges
- Type-erasure for heterogeneous event storage
- Thread-safe subscription management
- Preventing dangling callbacks (handle-based cleanup)

```cpp
// API design target
auto handle = EventBus::Subscribe<KeyPressEvent>([](const KeyPressEvent& e) {
    VYRO_INFO(INPUT, "Key pressed: {}", e.keycode);
});

EventBus::Publish<KeyPressEvent>({.keycode = Key::Space});
EventBus::Unsubscribe(handle);
```

#### Success Criteria
- Type-safe: wrong event type fails at compile time
- < 50ns dispatch overhead per event
- No memory leaks on unsubscribe

#### Industry Example

### Sub-Phase 1.6: Memory Management

#### Goal
Custom allocators that give predictable, fast, leak-free memory usage.

#### Duration: 6 days

#### Deliverables
- `PoolAllocator<T, BlockSize>` — fixed-size object pools
- `StackAllocator` — LIFO with O(1) reset
- `ArenaAllocator` — bulk allocation, reset together
- `FreeListAllocator` — general purpose with coalescing
- Debug wrappers (leak detection, use-after-free guards)
- `MemoryTracker` — global allocation statistics
- Aligned allocation utilities

#### Technical Challenges
- Alignment for SIMD (16B, 32B, 64B)
- Thread-safe pool allocator design
- Debug overhead without breaking release performance

#### Success Criteria
- Pool allocation: < 5ns per alloc
- Zero leaked allocations on shutdown (verified by MemoryTracker)
- ASAN clean

---

### Sub-Phase 1.7: Asset Management

#### Goal
A UUID-based asset registry that can load, cache, hot-reload, and track dependencies for all asset types.

#### Duration: 7 days

#### Deliverables
- `AssetManager` singleton with UUID-based lookup
- `AssetHandle<T>` — typed, reference-counted asset pointer
- `AssetLoader<T>` interface — extensible per asset type
- Asset metadata files (`.meta` files alongside assets)
- File watcher for hot reload (inotify / ReadDirectoryChangesW)
- Basic loaders: texture (stb_image), JSON
- Async loading (background thread pool)

#### Skills Required
- UUID generation (random 128-bit)
- Reference counting without shared_ptr overhead
- File system watching

#### Success Criteria
- Asset loaded once, cached on subsequent requests
- Hot reload triggers < 500ms after file change
- Dependency eviction: if A depends on B, reloading B reloads A

## Phase 2: ECS Architecture

### Goal
A production-quality archetype-based ECS that can handle 100,000+ entities at 60fps.

### Why It Is Needed
Every game system — rendering, physics, animation, scripting — operates on game objects. ECS is the engine's data backbone. Doing this wrong means performance problems that can never be fixed without a rewrite.

**Industry Example:** Unity DOTS uses archetype ECS. Flecs is the most complete open-source archetype ECS. Unreal uses its own UObject system.

### Estimated Duration
**6–8 weeks**

### Prerequisites
- Phase 1 complete (memory, events, logging)

---

### Sub-Phase 2.1: Entity Management

#### Goal
Fast entity creation/destruction with generational IDs.

#### Duration: 5 days

#### Deliverables
- `EntityID = uint64_t` (32-bit index | 16-bit generation | 16-bit flags)
- `EntityRegistry` — central entity store
- Generational ID validation (detect use-after-destroy)
- Entity creation: O(1) average
- Entity destruction: O(1) deferred or immediate mode
- Entity alive/dead check: O(1)

#### Success Criteria
- Create 1M entities in < 100ms
- Generational ID prevents stale entity access

---

### Sub-Phase 2.2: Component System

#### Goal
Cache-friendly component storage using archetype tables.

#### Duration: 10 days

#### Deliverables
- `ArchetypeTable` — contiguous arrays per component type per archetype
- `ComponentRegistry` — maps type → component ID at compile time
- `AddComponent<T>`, `RemoveComponent<T>`, `GetComponent<T>` operations
- Archetype migration (entity moves between tables when components change)
- Tag components (zero-size, no storage overhead)
- `Query<C1, C2, ...>().Exclude<C3>()` — archetype iteration

#### Technical Challenges
- Archetype migration without copying unnecessary data
- Query matching (bitset comparison on archetype signatures)
- Memory alignment for SIMD in component arrays

#### Success Criteria
- Iterate 100K entities with 3 components at > 500M components/sec
- Component add/remove: < 100ns per operation

---

### Sub-Phase 2.3: System Manager

#### Goal
Ordered, dependency-aware system execution with fixed and variable update rates.

#### Duration: 5 days

#### Deliverables
- `SystemManager` — registers and orders systems
- System priority / ordering tags
- Fixed-update systems (physics, deterministic)
- Variable-update systems (rendering, animation)
- System enable/disable at runtime
- System dependency declaration (A runs before B)

#### Success Criteria
- System execution order deterministic and correct
- System overhead < 1µs per frame (beyond system work)

---

### Sub-Phase 2.4: Scene Management

#### Goal
A scene graph layered on top of ECS, handling parent-child transforms and scene serialization.

#### Duration: 8 days

#### Deliverables
- `Scene` class — owns entity registry + systems
- `SceneGraph` — parent/child entity relationships via `RelationshipComponent`
- World transform propagation (dirty flags, lazy recompute)
- Scene serialization to JSON (entity + component data)
- Scene deserialization from JSON
- Prefab system (reusable entity templates)
- Scene switching and additive loading

#### Technical Challenges
- Transform hierarchy propagation at high entity counts
- JSON round-trip for all built-in components
- Prefab overrides (instance overrides parent prefab properties)

#### Industry Example
- Godot: node scene tree. Unity: GameObject hierarchy. Unreal: actor/component hierarchy.

#### Success Criteria
- Scene save/load with full fidelity (all components, all values)
- 10,000 parented entities transform update in < 1ms

---

### Phase 2 Skills Required
- Data-oriented design
- Cache-friendly data structures
- Template metaprogramming (type lists, compile-time reflection)
- JSON serialization (nlohmann/json)

### Phase 2 Success Criteria
- 100K entities with `Transform + Mesh + Velocity` at 60fps
- Scene serialization round-trip verified by test
- Benchmarks published and passing

## Phase 3: Rendering Engine

### Goal
A modular rendering pipeline supporting 2D sprites and 3D meshes, with a clean GAL that allows future Vulkan migration.

### Why It Is Needed
Rendering is what players see. Without a working renderer, no demo is possible, testing is impaired, and motivation collapses. Phase 3 produces the first visual output.

**Industry Example:** Godot's RenderingServer, Unity's SRP, Unreal's RHI.

### Estimated Duration
**12–16 weeks**

### Prerequisites
- Phase 1 complete (asset manager, memory)
- Phase 2 complete (ECS, scene)
- OpenGL knowledge or willingness to learn

---

### Sub-Phase 3.1: Graphics API Setup

#### Goal
Initialize a rendering window and get a triangle on screen.

#### Duration: 5 days

#### Deliverables
- GLFW / SDL3 window creation
- OpenGL 4.6 context initialization
- GLAD loader
- First triangle rendering
- Viewport resize handling
- SwapInterval / VSync support

#### Success Criteria
- Colored triangle renders at 60fps
- Window resize handled without crash

---

### Sub-Phase 3.2: Renderer Architecture

#### Goal
The GAL and render graph skeleton.

#### Duration: 10 days

#### Deliverables
- `IRenderDevice` interface (Buffer, Texture, Pipeline, Command buffer)
- `OpenGLRenderDevice` implementation
- `RenderGraph` — DAG of render passes
- `FrameData` — per-frame GPU data (UBOs, draw lists)
- Render thread separation (optional, deferred to optimization)

#### Success Criteria
- Render graph compiles pass dependencies
- OpenGL backend implements full IRenderDevice interface

---

### Sub-Phase 3.3: Shader System

#### Goal
Shader loading, compilation, hot reloading, and uniform binding.

#### Duration: 7 days

#### Deliverables
- `Shader` class (vert + frag + optional geom/compute)
- GLSL hot reload (file watcher integration)
- Uniform binding by name: `shader.SetFloat("time", t)`
- UBO binding blocks
- Shader include system (`#include "common.glsl"`)
- Shader cache (avoid recompilation)

#### Success Criteria
- Shader reload in < 200ms on file change
- All built-in uniforms bindable by name

---

### Sub-Phase 3.4: Texture System

#### Goal
Full texture loading, GPU upload, and sampling configuration.

#### Duration: 5 days

#### Deliverables
- `Texture2D` — RGBA8, RGB8, R8, HDR (RGBA32F)
- `CubemapTexture` — for skyboxes
- `TextureArray` — sprite sheet support
- Mipmap generation
- Sampler states (filter, wrap, anisotropy)
- Async GPU upload via staging buffer

#### Success Criteria
- PNG, JPG, HDR, KTX textures load correctly
- Mipmap chain generated automatically

---

### Sub-Phase 3.5: Camera System

#### Goal
Perspective and orthographic cameras with frustum culling.

#### Duration: 4 days

#### Deliverables
- `CameraComponent` (fov, near, far, aspect)
- Perspective and orthographic projection matrices
- View matrix from transform
- Frustum extraction from VP matrix
- Frustum-AABB culling (CPU-side)
- Primary camera selection

#### Success Criteria
- Off-screen objects not submitted for rendering (frustum culled)
- Camera move/rotate updates view matrix correctly

### Sub-Phase 3.6: 2D Renderer

#### Goal
Batched 2D sprite rendering for UI and 2D games.

#### Duration: 8 days

#### Deliverables
- `Renderer2D` — quad batching (up to 10,000 quads/batch)
- Texture atlas support
- Sprite component + system
- Z-ordering (depth sort)
- SDF font rendering (msdf-atlas-gen)
- Primitive drawing: line, rect, circle (debug + game use)

#### Success Criteria
- 10,000 sprites at 60fps in a single draw call (via batching)

### Sub-Phase 3.7: 3D Renderer

#### Goal
Full 3D mesh rendering with materials, lighting, and shadows.

#### Duration: 14 days

#### Deliverables
- `Mesh` — vertex buffer (pos, normal, tangent, uv, joints, weights)
- Assimp mesh importer (FBX, glTF, OBJ)
- `Material` — PBR-ready (albedo, roughness, metallic, normal, AO)
- Phong shading (immediate), PBR in Phase 10

- Basic shadow mapping (directional light only, MVP)

- `MeshComponent` + `LightComponent` ECS integration

#### Technical Challenges
- Assimp import pipeline (materials, textures, bones)
- Shadow mapping bias and peter-panning
- Multiple light batching in a forward renderer

#### Success Criteria
- glTF 2.0 model renders with correct materials and shadows
- 1000 draw calls at 60fps on minimum hardware

## Phase 4: Physics Engine

### Goal
A deterministic, integrated physics simulation with collision detection, rigid bodies, and constraints.

### Why It Is Needed
Games without physics feel lifeless. Physics is required for player controllers, projectiles, destructibles, and any dynamic simulation. Deterministic physics is required for network replay.

**Industry Example:** Unity uses PhysX. Godot uses Bullet (3D) and Box2D (2D). Unreal uses Chaos.

### Estimated Duration
**8–10 weeks**

### Prerequisites
- Phase 2 (ECS — `RigidBodyComponent`, `ColliderComponent`)
- Phase 3 (debug renderer for physics visualization)

---

### Sub-Phase 4.1: Collision Detection

#### Duration: 10 days

#### Deliverables
- Broadphase: Dynamic AABB tree (BVH)
- Narrowphase:
  - Sphere-Sphere
  - AABB-AABB
  - Sphere-AABB
  - OBB-OBB (SAT algorithm)
  - Convex-Convex (GJK algorithm)
  - Ray-Sphere, Ray-AABB, Ray-Mesh (for picking + physics queries)
- `ContactManifold` — contact points, normals, penetration depth

#### Success Criteria
- Broadphase culls 99%+ of pairs for 1000 objects
- GJK handles arbitrary convex shapes

---

### Sub-Phase 4.2: Collision Resolution

#### Duration: 7 days

#### Deliverables
- Sequential impulse solver (10 iterations default)
- Restitution (bounciness coefficient)
- Friction (Coulomb friction model)
- Baumgarte stabilization (position correction)
- Collision callbacks: `OnCollisionEnter`, `OnCollisionStay`, `OnCollisionExit`

#### Success Criteria
- Stacked boxes remain stable at 60fps
- Bouncing ball restitution correct

---

### Sub-Phase 4.3: Rigid Bodies

#### Duration: 5 days

#### Deliverables
- `RigidBody` — mass, inertia tensor, velocity, angular velocity
- Semi-implicit Euler integration
- Force/torque application
- Gravity (configurable per-world)
- Kinematic bodies (script-driven, no physics forces)
- Sleep/wake (stop simulating static objects)

#### Success Criteria
- Projectile trajectory matches analytical solution
- 1000 active rigid bodies at 60fps physics timestep

---

### Sub-Phase 4.4: Constraints

#### Duration: 6 days

#### Deliverables
- Distance constraint (rope-like)
- Hinge constraint (door/joint)
- Ball-socket constraint
- Slider constraint (prismatic)
- Motor (apply constant velocity to constraint)

#### Success Criteria
- Ragdoll assembled from hinge + ball-socket constraints
- Constraints do not explode under stress

---

### Sub-Phase 4.5: Physics Debugging

#### Duration: 3 days

#### Deliverables
- Debug draw: AABB wireframes, collision shapes, normals, velocities
- Integration with `Renderer2D` primitive drawing
- Toggle debug overlay at runtime (F3 key)

#### Success Criteria
- All collision shapes visualized in debug mode

---

### Phase 4 Skills Required
- Rigid body dynamics (Newton's laws, inertia tensors)
- GJK/EPA algorithms
- Constraint solving (sequential impulse method)

### Phase 4 Success Criteria
- Stable stacking, bouncing, ragdolls, constraints
- Deterministic: same input → same output always

## Phase 5: Audio Engine

### Goal
A full audio engine with spatial 3D audio, mixing, and resource streaming.

### Why It Is Needed
Audio is 50% of immersion. Silent games feel broken. A proper audio system handles music, SFX, ambient sound, and spatial positioning for 3D games.

**Industry Example:** Unity uses FMOD or their own audio engine. Unreal uses their own audio engine with MetaSounds. Godot uses its own audio bus system.

### Estimated Duration
**4–5 weeks**

### Prerequisites
- Phase 2 (ECS — `AudioSourceComponent`)
- Phase 1 (asset manager — `AudioAsset`)

---

### Sub-Phase 5.1: Audio Playback

#### Duration: 5 days

#### Deliverables
- miniaudio device initialization (output, 44100Hz/48000Hz stereo)
- `AudioClip` asset type (WAV, OGG, MP3)
- `AudioSource` — play, pause, stop, loop
- One-shot sounds vs persistent sources

#### Success Criteria
- WAV file plays on `AudioSource::Play()` call

---

### Sub-Phase 5.2: Audio Mixing

#### Duration: 5 days

#### Deliverables
- Audio bus system (Master → SFX / Music / Voice busses)
- Per-source volume, pitch, pan
- Bus volume control
- DSP effects chain: reverb, delay, lowpass filter

#### Success Criteria
- 32 concurrent audio sources mixed without glitching

---

### Sub-Phase 5.3: Spatial Audio

#### Duration: 5 days

#### Deliverables
- `AudioListener` — positioned in 3D space (camera-linked)
- Distance attenuation (linear, inverse, exponential)
- Doppler effect (velocity-based pitch shift)
- Panning based on listener-relative azimuth
- Occlusion (simple raycast-based volume reduction)

#### Success Criteria
- Moving audio source pans correctly relative to listener
- Distance attenuation audible at expected ranges

---

### Sub-Phase 5.4: Audio Resource Management

#### Duration: 4 days

#### Deliverables
- Streaming audio (large files read from disk incrementally)
- Cached audio (small SFX fully in memory)
- Audio memory budget enforcement
- Hot reload of audio clips

#### Success Criteria
- 100MB background music streams without full memory load
- Hot reload replaces audio clip within 300ms

---

### Phase 5 Skills Required
- Digital signal processing basics
- miniaudio API
- 3D audio mathematics

### Phase 5 Success Criteria
- Spatial audio sounds convincing in 3D scene
- 32+ simultaneous sounds without performance impact

## Phase 6: Animation System

### Goal
A complete skeletal animation system with blending and state machines.

### Why It Is Needed
Characters without animation are statues. Skeletal animation with blending enables smooth transitions between walk, run, jump, idle — the baseline for all character games.

**Industry Example:** Unity's Animator Controller, Unreal's Animation Blueprint, Godot's AnimationTree.

### Estimated Duration
**6–8 weeks**

### Prerequisites
- Phase 3 (mesh renderer, skinned mesh support)
- Phase 2 (ECS — `AnimatorComponent`)
- Phase 1 (asset manager — `AnimationClip` asset)

---

### Sub-Phase 6.1: Skeletal Animation

#### Duration: 10 days

#### Deliverables
- `Skeleton` — bone hierarchy (name, parent index, bind pose)
- `AnimationClip` — per-bone keyframes (pos, rot, scale) with timestamps
- Bone transform evaluation at time T (linear interpolation)
- GPU skinning (vertex shader with bone palette)
- Assimp animation import (FBX, glTF)
- `AnimatorComponent` holding current skeleton pose

#### Technical Challenges
- Quaternion slerp for rotation interpolation
- Bone palette upload to GPU (UBO or SSBO)
- Handling models with 200+ bones

#### Success Criteria
- Character model plays walk animation correctly
- GPU skinning runs at < 0.5ms for 10 skinned characters

---

### Sub-Phase 6.2: Animation Blending

#### Duration: 8 days

#### Deliverables
- 1D blend tree (blend between 2+ clips by float parameter)
- 2D blend tree (blend by 2D vector, e.g., move direction)
- Additive animation layers (e.g., aim overlay on top of walk)
- Blend mask (blend only upper/lower body)
- Cross-fade transition with configurable duration

#### Success Criteria
- Walk-to-run blend looks smooth at all blend weights
- Upper body aiming while lower body walks

---

### Sub-Phase 6.3: State Machines

#### Duration: 8 days

#### Deliverables
- `AnimationStateMachine` — states + transitions
- Transition conditions (bool, float, trigger parameters)
- Transition blending (cross-fade on transition)
- Nested state machines (sub-state machines)
- `AnimationParameters` — script-settable values
- JSON serialization of state machine graph

#### Industry Example
- Unity's Animator Controller is exactly this
- Unreal's Animation Blueprint state machine

#### Success Criteria
- Character transitions idle → walk → run → jump with correct blending
- State machine serializes and deserializes correctly

---

### Phase 6 Skills Required
- Skeletal animation theory (bind pose, inverse bind matrices, skinning)
- Quaternion mathematics (slerp, squad)
- State machine design patterns

### Phase 6 Success Criteria
- Animated character demo running in engine
- State machine-driven character controller functional
