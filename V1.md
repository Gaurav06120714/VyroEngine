## 1. Introduction

### 1.1 Purpose

This Software Requirements Specification defines the complete functional and non-functional requirements for **VyroEngine** — a custom game engine built from scratch using modern C++23 and Vulkan-first rendering. This document serves as the authoritative reference for all architectural decisions, feature prioritization, and development milestones.

### 1.2 Scope

VyroEngine is a full-stack game engine encompassing:
- Core runtime (ECS, memory, events, input)
- Rendering pipeline (2D + 3D, Vulkan + OpenGL)
- Physics simulation
- Audio engine
- Animation system
- Scripting layer (Lua)
- Visual editor (VyroEditor)
- Asset pipeline
- Networking stack

## 2. Vision

### 2.1 Core Vision Statement

VyroEngine is a **transparent, high-performance, cross-platform game engine** that gives developers complete architectural visibility and control. Unlike black-box engines, VyroEngine exposes every system through clean, documented APIs — from memory allocators to render graph nodes.

### 2.2 Design Philosophy

1. **Transparency Over Magic** — No hidden systems. Every behavior is explainable and overridable.
2. **Data-Oriented Design** — Cache-friendly, ECS-first architecture for maximum CPU throughput.
3. **Composition Over Inheritance** — Components compose behavior. Deep inheritance trees are forbidden.
4. **Zero-Cost Abstractions** — High-level APIs should compile down to the same assembly as hand-written C.
5. **Fail Loudly** — Assertions in debug builds. Silent failures are bugs.
6. **Deterministic Physics** — Reproducible simulation for networking and replay systems.

### 2.3 Inspirations

| Engine | What VyroEngine Borrows |
|--------|------------------------|
| **Godot** | Scene tree, node composition, MIT license spirit |
| **Unreal Engine** | Render pipeline quality, asset pipeline robustness |
| **Unity (DOTS)** | ECS architecture, burst compilation inspiration |
| **Hazel Engine** | Layered architecture, event system design |
| **The Cherno's Hazel** | Core engine layer design |
| **Bevy** | Rust ECS inspiration (translated to C++) |
| **id Tech** | Renderer design principles, BSP heritage |

## 3. Target Users

### 3.1 Primary Users

#### 3.1.1 Indie Game Developers
- Solo developers or small teams (1–5)
- Building 2D or 3D games
- Need: fast iteration, good tooling, reasonable learning curve
- Pain points with existing engines: licensing costs, black-box systems, runtime overhead

#### 3.1.2 Engine Students & Researchers
- Computer science students studying engine architecture
- Researchers needing a modifiable engine baseline
- Need: clean, documented, educational codebase

#### 3.1.3 Technical Game Developers (AAA Background)
- Experienced engineers who want full control
- Building custom rendering pipelines or gameplay systems
- Need: extensibility, no opinionated magic, raw performance access

### 3.2 Secondary Users

#### 3.2.1 Simulation Developers
- Non-game real-time simulations (architecture viz, training sims)
- Need: physics accuracy, real-time rendering, scripting

#### 3.2.2 Tooling Developers
- Developers building tools on top of VyroEngine
- Need: stable APIs, plugin system, editor extensibility

### 3.3 Anti-Target Users

- Enterprise developers needing fully supported, SLA-backed engines (use Unreal)
- Casual developers wanting drag-and-drop with zero code (use Unity/Godot)
- Console-first studios (until Phase 12+)

## 5. Programming Languages

### 5.1 Engine Core Language

**C++23** — Primary language for all engine systems.

| Requirement | Justification |
|-------------|--------------|
| C++23 standard | Concepts, ranges, std::expected, deducing this |
| Clang 17+ / MSVC 19.38+ / GCC 13+ | C++23 compliance |
| No exceptions in hot paths | Performance — use `std::expected` or error codes |
| No RTTI in hot paths | Performance — use custom type systems |

Key C++23 features used:
- `std::expected<T, E>` for error handling
- Deducing `this` for CRTP-free mixins
- `std::flat_map`, `std::flat_set` for cache-friendly containers
- Modules (experimental, opt-in)
- `std::print` for logging

### 5.2 Scripting Language

**Lua 5.4 / LuaJIT** — Game logic scripting.

| Requirement | Justification |
|-------------|--------------|
| Lua 5.4 | Stable, widely adopted, JIT-capable |
| LuaJIT 2.1 | 10–50x faster execution for game logic |
| sol3 binding layer | Modern, header-only C++/Lua binding |

### 5.3 Shader Language

| Backend | Shader Language |
|---------|----------------|
| Vulkan | GLSL 4.60 → SPIR-V |
| OpenGL | GLSL 4.60 |
| Metal (macOS) | MSL (transpiled via SPIRV-Cross) |
| WebGPU (future) | WGSL (transpiled via naga) |

### 5.4 Build Scripts

- **CMake 3.28+** — Cross-platform build system
- **Python 3.11+** — Asset pipeline scripts, code generation
- **Shell / Bash** — CI/CD pipelines

### 5.5 Editor Language

- **C++23** — Editor built as a separate module using the engine runtime
- **Lua** — Editor scripting and automation

## 6. Rendering APIs

### 6.1 Primary: Vulkan 1.3

| Aspect | Specification |
|--------|--------------|
| Version | Vulkan 1.3 |
| Extensions | VK_KHR_dynamic_rendering, VK_KHR_synchronization2, VK_EXT_mesh_shader |
| Shader compiler | glslc (shaderc) + SPIRV-Cross |
| Validation | Vulkan Validation Layers (debug builds) |
| Memory | VMA (Vulkan Memory Allocator) |

Vulkan is the primary rendering backend for all Tier 1 platforms. It provides:
- Explicit GPU control
- Multi-threaded command recording
- Ray tracing extension (future)
- Mesh shaders for geometry processing

### 6.2 Fallback: OpenGL 4.6

| Aspect | Specification |
|--------|--------------|
| Version | OpenGL 4.6 |
| Profile | Core profile |
| Extensions | ARB_bindless_texture, ARB_direct_state_access |
| Use case | Hardware without Vulkan support, debugging |

### 6.3 Platform-Specific

| Platform | Backend |
|----------|---------|
| macOS | Metal (via MoltenVK or native Metal backend) |
| WebAssembly | WebGPU |
| iOS | Metal |

### 6.4 Abstraction Layer

VyroEngine implements a **Graphics Abstraction Layer (GAL)** — a thin API over Vulkan/OpenGL/Metal:

```
GameCode → VyroRenderer API → GAL → [Vulkan | OpenGL | Metal | WebGPU]
```

The GAL provides:
- `CommandBuffer` abstraction
- `Pipeline` abstraction
- `Buffer` / `Texture` / `Sampler` resources
- Render pass and framebuffer management

## 13. Constraints & Assumptions

### 13.1 Constraints

1. **Solo developer initially** — Architecture must be achievable by one developer over 24 months
2. **Open source first** — MIT license; no proprietary dependency lock-in
3. **No game-specific code** — Engine is genre-agnostic
4. **No subscription models** — Engine is free to use and modify
5. **Standard C++ only** — No compiler-specific extensions in portable code

### 13.2 Assumptions

1. Developer has strong C++ knowledge (3+ years)
2. Developer understands linear algebra, quaternions, and 3D math
3. Target hardware supports Vulkan 1.1+
4. GitHub Actions provides sufficient CI/CD capacity
5. Open-source libraries remain maintained and compatible
